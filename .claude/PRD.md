# PRD: godot-bridge — CLI + Editor Plugin for AI-Assisted Godot Development

## Problem

Existing Godot AI integrations use MCP servers with 30-160+ tool definitions, consuming 6,000-30,000+ tokens of context per turn just for schema. We want a token-efficient alternative that gives coding agents (Claude Code, etc.) full control of the Godot 4.x editor through a CLI, paired with a lightweight editor plugin.

## Architecture

```
┌─────────────┐     bash        ┌──────────────┐   WebSocket    ┌──────────────┐
│ Coding Agent │───────────────►│ godot-bridge  │◄─────────────►│ Godot Plugin │
│ (Claude Code)│  (subprocess)  │ CLI (Go)      │  port 6505    │ (GDScript)   │
└─────────────┘                 └──────────────┘               └──────────────┘
       │                              │
       │ also reads/writes            │ file-based ops
       │ .tscn/.gd/.tres directly     │ need no WebSocket
       ▼                              ▼
   ┌──────────────────────────────────────┐
   │         Project Filesystem           │
   │  res://scenes/  res://scripts/  ...  │
   └──────────────────────────────────────┘
```

The agent discovers CLI commands via a `CLAUDE.md` snippet (generated by `godot-bridge reference`). One bash tool, not 42 MCP tool definitions.

## Components

### 1. Godot Editor Plugin (GDScript)

**Location in repo:** `godot-plugin/addons/godot_bridge/`

**What it does:** Runs inside the Godot 4.x editor. Starts a WebSocket server on `localhost:6505`. Receives JSON commands from the CLI and executes them using `EditorInterface` and the scene tree. Returns JSON responses.

**Plugin files:**
- `plugin.cfg` — standard Godot plugin manifest
- `plugin.gd` — `EditorPlugin` entry point, starts/stops the server
- `bridge_server.gd` — WebSocket server, command routing, JSON parsing

**Connection behavior:**
- Listens on `localhost:6505` (configurable via project setting `godot_bridge/port`)
- Single client connection at a time
- Heartbeat ping every 10 seconds
- Status indicator in editor bottom bar: "Bridge: Connected" / "Bridge: Listening"

**Command protocol:** JSON over WebSocket.

Request:
```json
{"id": "abc123", "command": "node_list", "args": {"path": "/root/Main", "depth": 2}}
```

Response:
```json
{"id": "abc123", "ok": true, "data": { ... }}
```

Error:
```json
{"id": "abc123", "ok": false, "error": "Node not found: /root/Main"}
```

**Supported commands the plugin handles** (things that REQUIRE the running editor):

| Command | Description |
|---------|-------------|
| `editor_state` | Returns: open scenes, selected nodes, current scene path, editor screen (2D/3D/Script) |
| `node_tree` | Get scene tree from a path, with configurable depth. Returns node names, types, key properties |
| `node_get` | Get full details of a single node: all properties, signals, groups, children |
| `node_add` | Add a node to the scene tree. Args: parent path, type, name, initial properties |
| `node_modify` | Set properties on an existing node. Args: node path, property dict |
| `node_delete` | Remove a node from the tree |
| `node_move` | Reparent or reorder a node |
| `scene_new` | Create a new scene file and open it. Args: `path` (res:// path to save), `root_type` (default `Node2D`), `root_name` (default: type name) |
| `scene_open` | Open a scene in the editor |
| `scene_save` | Save the current scene |
| `scene_run` | Run the current scene or the project (F5 / F6 equivalent) |
| `scene_stop` | Stop the running scene |
| `script_open` | Open a script in the script editor |
| `resource_list` | List files in a directory from the editor's filesystem dock |
| `screenshot` | Capture editor viewport as base64 PNG |

**Design principles for the plugin:**
- Minimal GDScript — thin wrapper around EditorInterface calls
- All commands support undo via `EditorUndoRedoManager` where applicable
- No AI/LLM logic in the plugin — it's a dumb command executor
- Respond with only essential data (don't dump every node property by default — use a `detail` param: `brief` / `full`)

### 2. CLI Tool (Go)

**Location in repo:** `cli/`

**Install:** Pre-built binaries or `go install`

**Executable name:** `godot-bridge`

**How it works:** Go CLI using a standard Go CLI library (e.g., `cobra` or `urfave/cli`). Each subcommand either (a) connects to the plugin's WebSocket to execute editor commands, or (b) operates directly on project files.

**Subcommands:**

```
godot-bridge reference          Print agent-optimized command reference (markdown)
godot-bridge status             Check if editor is connected, show project info

# Editor commands (require running Godot with plugin enabled)
godot-bridge editor state       Get editor state (open scenes, selection, etc.)
godot-bridge node tree [PATH]   Print scene tree from PATH (default: current scene root)
godot-bridge node get PATH      Get node details
godot-bridge node add TYPE --parent PATH --name NAME [--props '{"key":"val"}']
godot-bridge node modify PATH --props '{"position": [100, 200]}'
godot-bridge node delete PATH
godot-bridge node move PATH --new-parent PATH
godot-bridge scene new PATH --root-type Node2D --root-name Main  Create and open a new scene
godot-bridge scene open PATH    Open a scene file
godot-bridge scene save         Save current scene
godot-bridge scene run [PATH]   Run project or specific scene
godot-bridge scene stop         Stop running scene
godot-bridge script open PATH   Open script in editor
godot-bridge screenshot         Capture viewport, save to file, print path

# File commands (work without editor running)
godot-bridge file read PATH                Read a .gd, .tscn, .tres, or .cfg file
godot-bridge file write PATH --content STR Write content to a file
godot-bridge file list [DIR]               List project files, optionally filtered by directory
godot-bridge file search QUERY             Search file contents (grep-like)
```

**Output format:**
- Default: compact, human/agent-readable text (NOT verbose JSON)
- `--json` flag: structured JSON output for programmatic use
- Errors go to stderr, data goes to stdout

**Connection handling:**
- On any editor command: attempt WebSocket connect to `localhost:6505`
- If connection fails: print clear error "Godot editor not running or plugin not enabled" and exit 1
- Timeout: 5 seconds for connection, 30 seconds for command response

### 3. Agent Context: `reference` Command + CLAUDE.md

`godot-bridge reference` outputs a compact markdown block (~60-80 lines) that serves as the agent's command reference. This same content is suitable for inclusion in a project's `CLAUDE.md`.

Example output:

```markdown
# godot-bridge CLI Reference

## Connection
- Requires Godot 4.x editor running with godot_bridge plugin enabled
- Plugin listens on localhost:6505
- `godot-bridge status` to verify connection

## Commands

### Editor State
godot-bridge editor state              # open scenes, selected nodes, active screen

### Scene Tree
godot-bridge node tree [/root/Main]    # tree from path (default: scene root)
godot-bridge node get /root/Main/Player  # full node details
godot-bridge node add Sprite2D --parent /root/Main --name Hero
godot-bridge node modify /root/Main/Player --props '{"position":[100,200]}'
godot-bridge node delete /root/Main/TempNode
godot-bridge node move /root/Main/A --new-parent /root/Main/SubTree

### Scenes & Scripts
godot-bridge scene open res://scenes/level1.tscn
godot-bridge scene save
godot-bridge scene run                 # F5 equivalent
godot-bridge scene stop
godot-bridge script open res://scripts/player.gd

### Files (no editor needed)
godot-bridge file read res://scripts/player.gd
godot-bridge file write res://scripts/enemy.gd --content '...'
godot-bridge file list res://scenes/
godot-bridge file search "class_name Player"

### Screenshots
godot-bridge screenshot                # saves PNG, prints file path

## Tips
- File ops (read/write/list/search) work without the editor — use them freely
- `node tree` with no args shows the whole current scene
- Properties in --props use JSON: vectors as arrays [x,y], colors as [r,g,b,a]
- All editor commands support --json for structured output
```

### 4. CLAUDE.md Integration

The repo includes a `CLAUDE.md.example` that users copy into their Godot project root. It contains:

1. The `reference` output (command cheat sheet)
2. Brief Godot conventions (file structure, GDScript style, .tscn format basics)
3. Guidance like "prefer file ops over editor commands when possible — they're faster and don't require the editor"

## Design Decisions

**Why CLI over MCP:**
- Single bash tool definition (~50 tokens) vs 15-40 MCP tool definitions (~3,000-8,000 tokens)
- Agent can batch: `godot-bridge node tree && godot-bridge file read res://scripts/player.gd` in one bash call
- Shell pipes and scripting for free
- No JSON-RPC framing overhead per call

**Why WebSocket (not HTTP):**
- Persistent connection — avoids reconnect overhead on every command
- Bidirectional — future: editor can push events (errors, scene changes) to CLI
- Matches what existing Godot MCP plugins use, so the plugin pattern is proven

**Why Go for CLI:**
- Single static binary — easy to distribute, no runtime dependencies
- Fast startup time — important for CLI tools used in tight agent loops
- Strong standard library with good WebSocket support
- Cross-platform compilation to macOS, Linux, Windows from one codebase

**Why NOT parsing .tscn files in the CLI:**
- Phase 1: `file read` just returns raw file content. The agent (LLM) is good at understanding .tscn format.
- Phase 2 (future): add a `scene parse` command that returns structured data from .tscn files if raw parsing proves too token-heavy.

**Property encoding:**
- GDScript types map to JSON: `Vector2(100, 200)` → `[100, 200]`, `Color(1,0,0,1)` → `[1,0,0,1]`
- The plugin handles conversion between JSON arrays and Godot types
- The CLI passes through JSON strings without interpreting them

## Implementation Priorities

1. **Phase 1 — Minimum viable bridge**
   - Plugin: WebSocket server, `editor_state`, `node_tree`, `node_get`, `scene_save`, `scene_open`
   - CLI: `status`, `editor state`, `node tree`, `node get`, `scene open`, `scene save`, `file read`, `file write`, `file list`, `reference`
   - CLAUDE.md.example

2. **Phase 2 — Full node manipulation**
   - Plugin: `node_add`, `node_modify`, `node_delete`, `node_move`, undo/redo support
   - CLI: corresponding subcommands
   - `screenshot` command

3. **Phase 3 — Run/debug**
   - Plugin: `scene_run`, `scene_stop`, error capture from running game
   - CLI: `scene run`, `scene stop`
   - Event push from editor to CLI (compile errors, runtime errors)

## Compatibility

- Godot 4.3+
- Go 1.22+
- Tested on macOS, Linux, Windows
